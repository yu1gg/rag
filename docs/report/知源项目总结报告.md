## 项目总结报告

> **项目名称**：知源（ZhiYuan）— 基于检索增强生成的智能问答与文本摘要系统  
> **开发周期**：2026年6月  
> **技术栈**：Python FastAPI + Vue 3 + TypeScript + FAISS + BGE-M3  
> **项目性质**：深度学习应用实践课程项目

## 一、项目概述

### 1.1 项目定位

「知源」是一套基于检索增强生成（Retrieval-Augmented Generation, RAG）技术的智能知识工作台。系统允许用户输入自然语言问题，从本地知识库中检索相关文档片段，结合大语言模型（LLM）生成带来源标注的答案。同时支持对长文本进行结构化摘要压缩。

### 1.2 核心价值

RAG 技术解决了大语言模型三大痛点：

| 痛点 | RAG 解决方案 |
|------|-------------|
| **知识截止日期** — 模型训练数据有截止时间 | 知识可随时更新，只需重建索引，无需重新训练模型 |
| **幻觉问题** — 模型可能编造信息 | 每个答案基于检索到的真实文档生成，大幅降低编造概率 |
| **不可追溯** — 用户无法判断信息来源 | 每个答案附带完整的参考来源（文档标题、来源类型、原文链接） |

### 1.3 功能概览

1. **智能问答（QA）**：输入问题 → 检索知识库 → LLM 生成答案 → 右侧展示批注来源，支持点击原文链接跳转
2. **文本摘要（Summary）**：上传或粘贴长文本 → 自动提取核心主题、关键要点、详细摘要，结构化展示
3. **批注来源追溯**：每个答案的引用片段可点击弹出详情卡片，显示文档标题、来源类型、日期、原文链接
4. **双检索策略**：支持向量检索（FAISS 语义匹配）和关键词检索（BM25 精确匹配）两种模式，用户可按需切换
5. **流式输出**：答案逐字实时输出，提升交互体验
6. **多轮对话记忆**：自动携带最近 6 轮会话上下文，支持追问和深入讨论
7. **文档上传**：支持上传 .txt / .md 文件进行摘要分析

## 三、核心原理与技术实现

### 3.1 RAG 检索增强生成流程

```
用户问题
  │
  ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. 检索   │ → │ 2. 重排   │ → │ 3. 上下文  │
│ FAISS/BM25│    │ CrossEnc  │    │ 构建      │
│ 向量/关键词│    │ 精确重排  │    │ 裁剪+拼接  │
└──────────┘    └──────────┘    └──────────┘
                                      │
                                      ▼
                                 ┌──────────┐    ┌──────────┐
                                 │ 4. Prompt │ → │ 5. LLM   │
                                 │ 组装      │    │ 生成答案  │
                                 │ 模板+历史  │    │ 流式/非流 │
                                 └──────────┘    └──────────┘
```

### 3.2 向量检索（FAISS）

**原理**：使用 BGE-M3 嵌入模型将文本转换为 1024 维向量。查询时，同样将用户问题编码为向量，在 FAISS `IndexFlatIP` 索引中求内积（等价于余弦相似度），返回 Top-K 最相关文档片段。

**优势**：
- 语义匹配能力强 — "循环神经网络" 能匹配到 "RNN" 相关文档
- BGE-M3 支持中英文混合检索，无需分词
- FAISS 精确搜索，108 个文档中毫秒级响应

**局限**：
- 对专有名词、缩写、代码片段的精确匹配可能不如关键词检索
- 嵌入模型加载需要约 1.5GB 显存/内存

### 3.3 关键词检索（BM25）

**原理**：使用 `rank_bm25` 库实现 BM25Okapi 算法。前端用 jieba 分词处理中文文本，构建倒排索引。查询时统计每个 chunk 中查询词的词频（TF）× 逆文档频率（IDF），加权求和得到相关性分数。分数归一化到 0-1 区间，低于 0.1 的弱匹配自动过滤。

**优势**：
- 精确关键词命中 — "attention" 只匹配包含 "attention" 的文档
- 对专有名词、缩写、代码标识符敏感
- 索引构建快（纯内存操作，无需 GPU），轻量零依赖

**局限**：
- 无法理解同义词和语义变体
- 中文分词精度影响检索效果
- 小语料（<10 文档）中 IDF 退化

### 3.4 向量检索 vs 关键词检索对比

| 维度 | 向量检索（FAISS） | 关键词检索（BM25） |
|------|-------------------|---------------------|
| **匹配原理** | 语义向量相似度 | 词频-逆文档频率 |
| **中文同义词** | 优秀 | 差 |
| **专有名词** | 一般 | 优秀 |
| **代码/缩写** | 一般 | 优秀 |
| **长查询** | 好 | 一般 |
| **计算资源** | 需要 GPU/内存加载嵌入模型 | 纯 CPU，轻量 |
| **检索速度** | 毫秒级（编码+搜索） | 毫秒级（纯计算） |
| **适用场景** | 自然语言问题、释义查询 | 精确术语搜索、代码查询 |

### 3.5 流式输出（SSE）

**原理**：LLM 的 `stream=True` 模式逐 token 返回文本增量。后端使用 FastAPI `StreamingResponse` 包装为 SSE（Server-Sent Events）格式。前端使用 `ReadableStream` 逐行解析，实时更新聊天消息文本。

**关键设计**：流式模式下不做应用层重试（stream 不可回放）；设置 `X-Accel-Buffering: no` 防止反向代理缓冲。

### 3.6 Markdown 渲染

**原理**：使用 `marked` 库将 LLM 返回的 Markdown 格式文本（标题、列表、代码块、表格、粗体等）渲染为 HTML，通过 Vue 的 `v-html` 指令安全注入展示。

## 四、数据处理管道

### 4.1 五源数据采集

| 采集器 | 数据源 | 标识 |
|--------|--------|------|
| `ArxivCollector` | arXiv 学术论文摘要（通过 arxiv Python SDK） | `arxiv` |
| `NewsCollector` | Hacker News API 热门技术新闻 | `news` |
| `CourseMaterialCollector` | 本地课程资料（.md/.txt 文件） | `course` |
| `QAPairCollector` | 自定义问答对（手动编写 + 模板生成） | `qa` |
| `DeepSeekDocsCollector` | DeepSeek API 官方文档（Web 爬取） | `deepseek_docs` |

每篇文档自带完整元数据：`title`（标题）、`source`（来源类型）、`url`（原文链接）、`date`（日期）。

### 4.2 数据处理流程

```
5 源采集 → TextCleaner（HTML/空白/控制字符清洗）
         → Deduplicator（MinHash 指纹去重，相似度阈值 0.9）
         → FilterShort（过滤<20字符）
         → Chunker（512 token滑动窗口，128 overlap，段落边界感知）
         → Embedder.encode（BGE-M3 批量编码 32条/批）
         → VectorStore.add（FAISS IndexFlatIP + JSON metadata）
```

## 六、创新点

### 6.1 双检索策略

**创新描述**：同时支持向量检索（FAISS）和关键词检索（BM25），用户可按需切换。两种检索返回相同数据格式，下游处理完全复用。

**应用场景示例**：
- 问 "DeepSeek 怎么开启思考模式？" → 向量检索（语义理解"思考模式"和 "thinking mode"）
- 问 "deepseek-v4-pro max_tokens 参数怎么设？" → 关键词检索（精确匹配 "max_tokens"）

### 6.2 全链路元数据追溯

**创新描述**：Document → Chunk → VectorStore → SearchResult → API Response → 前端 ReferenceItem，8 个环节逐层传递文档元数据（标题、来源、URL、日期）。用户点击每个引用片段可以看到「这段内容来自哪篇文档、哪个来源、原文链接是什么」。

**技术价值**：将 RAG 系统从"黑盒答案机"升级为"可审计知识系统"。

### 6.3 流式 SSE 实时输出

**创新描述**：LLM 答案逐 token 实时推送，前端通过异步生成器（`AsyncGenerator`）消费 SSE 流，无需第三方库。文本逐字出现在聊天气泡中，消除用户等待焦虑。

### 6.4 Embedding LRU 缓存

**创新描述**：在 `encode_query()` 中实现手写 LRU 缓存（128 条，FIFO 淘汰），多轮对话和重复查询中能避免重复编码。无需引入 Redis 等外部依赖。

### 6.5 模型降级容错

**创新描述**：嵌入模型缺失时自动切换到确定性哈希散列向量（1024 维 MD5 散列 × L2 归一化），重排器加载失败时自动切换到透传模式（结果原样返回）。确保系统不会因为模型文件缺失而崩溃。

### 6.6 结构化摘要展示

**创新描述**：摘要调用分析 LLM 返回的 `【核心主题】【关键要点】【详细摘要】` 结构，前端自动解析并分别展示在右侧面板中——聊天气泡只显示详细摘要，面板中额外展示核心主题卡片和关键要点列表。

## 七、遇到的困难与解决方案

### 7.1 BM25 关键词检索分数显示异常

**问题**：前端将 BM25 原始分数（无上限，如 19.29）乘以 100 显示为百分比，出现 "1929%" 等荒谬值。

**解决**：在 `KeywordRetriever.retrieve()` 中对分数做归一化处理（除以最高分），将结果映射到 0-1 区间，与向量检索分数口径一致。同时过滤归一化后低于 0.1 的弱匹配噪声。

### 7.2 流式输出中途截断

**问题**：长回答流式输出中途被切断，用户看到半句话。

**根因分析**：
1. `OPENAI_TIMEOUT=20` 秒 — 流式输出持续时间超过 20 秒时连接被断
2. `stream_options={"include_usage": True}` — 小米 MiMo API 不支持此参数
3. `QA_MAX_TOKENS=512` — token 数量不足

**解决**：
1. 将 `OPENAI_TIMEOUT` 提升至 120 秒
2. 移除 `stream_options` 参数，恢复纯 `stream=True` 通用兼容写法
3. 将 `QA_MAX_TOKENS` 提升至 2048

### 7.3 索引构建时嵌入模型缺失

**问题**：6月8日首次构建索引时 BGE-M3 模型不在本地缓存，`build_index` 使用了 hash fallback 嵌入（无语义），导致查询时语义匹配失效。

**解决**：安装 BGE-M3 模型到本地缓存后，运行 `python -m backend.scripts.build_index` 重建索引。验证：查询 "RNN LSTM GRU" → 修复前 chunk_000083 不在 Top-5；修复后 chunk_000083 Rank 0，相似度 0.65。

### 7.4 DeepSeek 中文文档编码乱码

**问题**：`BeautifulSoup(response.text)` 在 Windows 环境下默认用 GBK 解码 UTF-8 编码的网页内容，导致所有中文字符变成乱码。

**解决**：在传给 BeautifulSoup 前显式设置 `response.encoding = "utf-8"`，然后重建全管线。

### 7.5 小语料 BM25 IDF 退化

**问题**：单元测试中使用 2-4 个文档的微型语料，ATIRE 公式下 `log(N-df+0.5)-log(df+0.5)` 导致 IDF=0，全部 BM25 分数为 0。

**解决**：将单元测试语料扩展到 15-20 个文档，使 IDF 恢复正常。

## 八、应用场景

### 8.1 企业内部知识库问答

部署专属知识库（产品手册、技术文档、规章制度），员工用自然语言提问获取精确答案。每个答案可追溯到源文档，支持审计核查。

### 8.2 课程学习辅助

上传课程教材、讲义、参考资料 → 构建知识索引 → 学生提问获取基于课程内容的标准答案。本项目的 DeepSeek API 文档知识库就是典型应用。

### 8.3 技术文档智能助手

类似于本项目中的 DeepSeek 文档爬取，对开源项目、API 文档、开发者指南建立 RAG 索引，帮助开发者快速查找使用方法、参数说明、代码示例。

### 8.4 长文本智能处理

上传研究报告、论文、新闻稿 → 自动提取核心主题和关键要点，节约阅读时间。摘要结构化的输出方便快速浏览和归档。

## 九、技术选型理由

| 技术选型 | 理由 |
|----------|------|
| **FastAPI** | 原生异步支持、自动生成 OpenAPI 文档、类型安全的请求校验 |
| **Vue 3 Composition API** | 无需额外状态管理库（无 Pinia/Axios），`ref`/`computed` 足以管理所有状态 |
| **FAISS** | Meta 出品，高性能精确搜索，支持内积/余弦相似度，本地部署无网络依赖 |
| **BGE-M3** | 多语言嵌入模型，中英文表现优异，1024 维向量，支持本地加载 |
| **rank-bm25** | 轻量纯 Python BM25 实现，零外部依赖 |
| **marked** | 轻量 Markdown→HTML 转换器，支持 GitHub Flavored Markdown |
| **纯手写 CSS** | 无 UI 框架依赖，设计系统通过 CSS 变量管理，组件样式完全可控 |
