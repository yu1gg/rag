# project_X 后续优化计划

> 日期：2026-06-12  
> 基线：`master` (d15157f)  
> 定位：课程项目收尾展示 + 响应速度提升  
> 参考：`docs/项目架构与维护说明.md` 第5.3节（已知局限）、第6节（优化建议）及代码审计报告

## 一、审计发现汇总

### 文档已知局限（第5.3节）
1. 没有数据库
2. 没有鉴权
3. 没有任务队列
4. 没有流式输出
5. 没有知识图谱
6. 没有服务端多轮会话记忆
7. 前端会话只做本地持久化
8. `evaluations/latest` 是读结果接口，不是全套评估平台

### 文档优化建议（第6.1节）
1. 流式输出（SSE/WebSocket）
2. 重操作后台任务化
3. 更完整的评测链路
4. 更细粒度的前端详情视图
5. 状态监控与日志观测

### 代码审计新增问题
1. 没有后端会话历史/多轮对话记忆（高）
2. 没有流式输出支持（中）
3. 前端零测试（中）
4. `@app.on_event("startup")` 已弃用（中）
5. `async def` 路由调用阻塞代码（低）
6. 缺少 catch-all 404 路由（低）
7. `http.ts` 网络错误未捕获（低）
8. `api_prefix` 硬编码（低）
9. `fetch()` 无请求超时（低）
10. `time.sleep` 阻塞事件循环（低）
11. 无 embedding 查询缓存（低）
12. `error.ts` 未被充分使用（低）

## 二、优化优先级（按展示效果 + 响应速度排序）

### 阶段一：展示体验 + 响应速度

| # | 优化项 | 来源 | 价值 |
|---|--------|------|------|
| 1 | 流式输出 (SSE) | 文档6.1 / 审计#2 | 演示效果提升最大的单项改动，用户逐字看到回答 |
| 2 | 多轮对话记忆 | 审计#1 | 让演示像真正的聊天——追问、反问、深入讨论 |
| 3 | LLM 响应缓存 | 审计#11 | 相同 query 在 session 内重复时直接返回缓存 |
| 4 | embedding 查询缓存 | 审计#11 | LRU 缓存已有 query embedding，减少重复编码 |
| 5 | LLM 调用优化 | 审计#10 | `time.sleep` 改异步化或线程池，减少首字延迟 |

### 阶段二：工程稳健性

| # | 优化项 | 来源 | 价值 |
|---|--------|------|------|
| 6 | `@app.on_event` → lifespan | 审计#4 | FastAPI 新 API，避免未来不兼容 |
| 7 | 前端网络错误处理 | 审计#7/9/12 | `http.ts` 加超时、网络错误捕获、`error.ts` 接入 |
| 8 | 前端 404 路由 | 审计#6 | 访问不存在路径不再白屏 |
| 9 | `api_prefix` 配置化 | 审计#8 | 硬编码 `/api/v1` 移入 `.env` |
| 10 | async route 修正 | 审计#5 | `async def` 但无 `await` 的路由改为 `def` |

### 阶段三：体验打磨（有余力时做）

| # | 优化项 | 来源 | 价值 |
|---|--------|------|------|
| 11 | 点选消息回看详情 | 文档6.1#4 | 右侧详情栏支持点击历史消息回看 |
| 12 | 结构化日志 + 耗时统计 | 文档6.1#5 | 当前字符串拼接改为 JSON 结构化日志 |
| 13 | 后端脚本 Web 触发 | 文档6.1#2 | `build_index` 等脚本增加简单 Web 触发入口 |
| 14 | 前端基础测试 | 审计#3 | 核心组件 vitest 测试 |

### 暂不纳入

| 项目 | 原因 |
|------|------|
| 数据库引入 | 文档6.2 明确不建议"为了看起来完整而先引入数据库" |
| 鉴权/账号系统 | 文档6.2 明确不建议"为了像聊天产品而先做复杂账号体系" |
| 知识图谱 | 超出课程项目范围 |
| 任务队列 | 当前脚本量级不需要 Celery/RQ |
| 评测平台化 | 当前 eval 脚本输出满足课程要求 |

## 三、建议执行顺序

```
阶段一（优先）：
  #1 流式输出 → #2 多轮对话 → #4 embedding缓存 → #5 LLM优化

阶段二：
  #6 lifespan → #7 错误处理 → #8 404 → #9 配置化 → #10 async修正

阶段三：
  按需挑做
```

## 四、约束

- 保持现有架构边界：后端壳层薄、服务层厚、底层模块职责单一
- 不引入数据库、不引入鉴权
- 前端坚持单工作台交互模型
- 配置只走 `.env`，数据只走 `backend/storage/`
- 阶段三各项仅在有余力时启动，不阻塞阶段一二
