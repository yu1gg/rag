# 项目总体框架设计

> 课题：基于大模型的智能问答与文本摘要应用
> 课程：深度学习应用实践
> 日期：2026-06-08

---

## 一、项目概述与目标

### 1.1 课题背景

本项目采用**检索增强生成（RAG）**架构，构建一个基于大语言模型的智能问答与文本摘要应用。系统通过离线阶段完成文档分块、向量嵌入与知识库索引构建，在线阶段将用户问题向量化后检索相关文本块，拼接上下文注入提示词，由大模型生成高质量回答或摘要。

### 1.2 核心目标

1. **智能问答**：基于检索到的课程资料、论文等知识，以课程助教角色回答用户问题
2. **文本摘要**：对长文本进行结构化摘要，控制长度为原文15%-30%
3. **知识问答**：基于检索材料作答，自动标注引用来源
4. **完整评估**：自动评估与人工评估相结合，输出指标报告与案例分析

### 1.3 三大功能模式

| 模式 | 角色设定 | 核心能力 | 特殊要求 |
|------|---------|---------|---------|
| 智能问答 | 课程助教 | 基于检索材料回答问题 | 不相关时告知超出范围 |
| 文本摘要 | 摘要助手 | 分步摘要（关键句→归纳→结构化） | 长度15%-30% |
| 知识问答 | 知识专家 | 基于资料作答 | 标注引用来源 |

## 三、数据流设计

### 3.1 离线阶段（索引构建，一次性执行）

```
                    采集               清洗               分块
原始数据源 ──────▶ raw/*.json ─────▶ 清洗后文本 ─────▶ chunks.json
(arXiv/新闻/课程)                                     
                                                          │
                                                          ▼
用户查询 ◀─── 生成结果 ◀── LLM ◀── 提示词 ◀── 重排结果 ◀── FAISS检索 ◀── 向量化 ◀── BGE-M3嵌入
                                                          ▲
                                                          │
                                                   faiss.index (持久化)
```

**流程说明：**

1. **数据采集**：从 arXiv API、新闻公开 API、本地课程资料、手工 QA 对四个来源采集数据，统一以 JSON 格式存储
2. **文本清洗**：去除 HTML 标签、特殊字符，统一 UTF-8 编码，MinHash 去重，过滤无效短文本
3. **文本分块**：采用固定长度滑动窗口法（chunk_size=512 tokens, overlap=128 tokens），段落边界优先切分
4. **向量嵌入**：使用 BGE-M3 模型将每个文本块转换为 1024 维归一化向量
5. **索引构建**：将向量存入 FAISS IndexFlatIP 索引，持久化到磁盘

### 3.2 在线阶段（每次查询）

```
用户输入 ──▶ Query Embedding ──▶ FAISS 检索 Top-K ──▶ Cross-Encoder 重排
                                                              │
                                                              ▼
结果展示 ◀── 参考片段提取 ◀── LLM 生成 ◀── 提示词填充 ◀── 上下文拼接
```

**流程说明：**

1. **Query 向量化**：将用户问题通过 BGE-M3 转换为向量
2. **Top-K 检索**：在 FAISS 索引中进行余弦相似度检索，默认 K=5
3. **Cross-Encoder 重排序**：使用 bge-reranker-v2-m3 对 Top-K 结果重新排序，提升相关性
4. **上下文拼接**：将重排后的文本块拼接为上下文，注入提示词模板
5. **LLM 生成**：调用 DeepSeek/MiMo API 生成回答
6. **结果返回**：生成答案 + 参考文档片段 + 来源标注

## 四、各层详细设计

### 4.1 数据层

#### 4.1.1 数据采集

| 数据源 | 采集方式 | 目标量 | 存储字段 |
|--------|---------|--------|---------|
| arXiv 论文摘要 | `arxiv` Python 包 | ≥20篇 | id, title, content, source, url, date |
| 课程资料 | 本地 Markdown/TXT/PDF 读取 | ≥15篇 | id, title, content, source, chapter |
| 公开新闻 | `requests` + 公开 API | ≥15篇 | id, title, content, source, date |
| 自建中文 QA 对 | 手工编写 + 模板辅助生成 | ≥200组 | id, question, answer, source |

- **统一输出格式**（JSON）：

```json
{
  "id": "doc_001",
  "title": "文档标题",
  "content": "文档正文内容...",
  "source": "arxiv",
  "url": "https://arxiv.org/abs/xxxx.xxxxx",
  "date": "2024-01-15",
  "char_count": 3500
}
```

- 采集脚本带进度显示（tqdm）和异常重试（最多 3 次）
- 总量要求：文档级文本 ≥50篇，总字符数 ≥10万字

#### 4.1.2 文本清洗

| 步骤 | 方法 | 说明 |
|------|------|------|
| HTML 标签移除 | BeautifulSoup | 解析 HTML，提取纯文本 |
| 特殊字符规范化 | 正则 + Unicode 映射 | 全角→半角，统一空白符 |
| 编码统一 | UTF-8 | 强制转码 |
| 段落去重 | MinHash LSH | 相似度阈值 0.9，保留一份 |
| 无效短文本过滤 | 长度检查 | < 20 字符直接剔除 |

#### 4.1.3 文本分块

- **算法**：固定长度滑动窗口法
- **参数**：chunk_size=512 tokens, overlap=128 tokens
- **段落边界优先**：优先在 `\n\n` 位置切分，避免截断句子
- **Token 计数**：使用 `tiktoken` 精确计算
- **输出结构**：

```json
{
  "chunk_id": "chunk_001",
  "doc_id": "doc_001",
  "content": "分块文本内容...",
  "char_start": 0,
  "char_end": 1024,
  "token_count": 512
}
```

### 4.2 嵌入层

#### 4.2.1 向量化

- **模型**：BGE-M3 (`BAAI/bge-m3`)
- **加载方式**：`sentence-transformers` 库
- **向量维度**：1024
- **归一化**：L2 归一化后存入 FAISS（使得内积等价于余弦相似度）
- **批处理**：batch_size=32
- **设备**：自动检测 GPU，无 GPU 时回退 CPU

```
encode("文本块内容") → [0.012, -0.034, ..., 0.008]  (1024维归一化向量)
```

#### 4.2.2 FAISS 索引

- **索引类型**：`IndexFlatIP`（精确内积搜索）
- **元数据管理**：`chunk_meta.json` 存储 chunk_id ↔ 文本块内容的映射
- **持久化文件**：
  - `data/index/faiss.index` — 向量索引文件
  - `data/index/chunk_meta.json` — 元数据映射文件

### 4.3 检索层

#### 4.3.1 Top-K 检索

- **输入**：用户查询文本
- **流程**：Query → BGE-M3 向量化 → FAISS 余弦检索 → Top-K 结果
- **默认 K=5**，可通过配置文件调整
- **输出**：`[{chunk_id, content, score}]`

#### 4.3.2 Cross-Encoder 重排序

- **模型**：`BAAI/bge-reranker-v2-m3`
- **加载方式**：`FlagEmbedding` 库
- **输入**：(query, candidate_chunks) 对
- **输出**：按相关性重新排序的结果列表
- **效果**：在 Top-K 基础上进一步提升检索精度

### 4.4 生成层

#### 4.4.1 提示词模板

**模板一：智能问答（课程助教）**

```
你是一位专业、耐心的课程助教。请根据以下参考资料回答用户的问题。
要求：
1. 如果参考资料足以回答，请给出准确、完整的答案
2. 如果参考资料不足以回答，请明确告知"该问题超出了我目前的知识范围"
3. 回答应清晰易懂，适合学生理解

参考资料：
{context}

用户问题：{question}
```

**模板二：文本摘要**

```
请按照以下步骤对文本进行摘要：

第1步 - 提取关键句：识别原文中最能体现核心内容的3-5个句子
第2步 - 归纳概括：将关键信息提炼为简洁的概括性表述
第3步 - 结构化输出：
  - 【核心主题】：一句话概括全文主旨
  - 【关键要点】：以要点形式列出3-5个核心观点
  - 【详细摘要】：连贯的摘要段落

摘要总长度应控制在原文的15%-30%之间。

原文：
{context}
```

**模板三：知识问答（引用标注）**

```
请根据以下参考资料回答用户的问题。每个参考资料的编号格式为[来源N]。
要求：
1. 基于参考资料给出准确答案
2. 在答案中标注所引用的来源编号，例如"根据[来源1]和[来源3]..."
3. 如果参考资料中不包含相关信息，请明确说明

参考资料：
{context}

用户问题：{question}
```

#### 4.4.2 LLM Client

- **接口协议**：OpenAI 兼容接口（`/v1/chat/completions`）
- **支持模型**：
  | 提供商 | base_url | model |
  |--------|----------|-------|
  | DeepSeek | `https://api.deepseek.com/v1` | `deepseek-chat` |
  | MiMo | 待配置 | `mimo-v2.5-pro` |
- **参数**：temperature（可配置，默认 0.7），max_tokens（可配置，默认 2048）
- **异常处理**：API 超时重试 3 次，指数退避
- **长文本摘要策略**：超长文本 → 分段调用摘要模板 → 合并摘要 → 二次归纳

### 4.5 评估层

#### 4.5.1 自动评估指标

| 指标 | 方法 | 库 | 说明 |
|------|------|-----|------|
| ROUGE-L | 最长公共子序列 | `rouge-score` | 摘要与参考摘要的重叠度 |
| BLEU | n-gram 匹配 | `sacrebleu` | 1-gram 到 4-gram 匹配 |
| BERTScore | 语义向量相似度 | `bert-score` | 基于 BERT 的语义相似度 |
| Hit@K | 检索命中率 | 自定义 | 检索结果中是否包含正确答案 |

#### 4.5.2 人工评估

- **样本**：选取 20 组典型问题
- **评分维度**（各 5 分制）：

| 维度 | 1分 | 3分 | 5分 |
|------|-----|-----|-----|
| 正确性 | 答案错误 | 部分正确 | 完全准确 |
| 完整性 | 大量遗漏 | 基本覆盖 | 全面完整 |
| 流畅性 | 语句不通 | 基本通顺 | 自然流畅 |

- **案例分类**：成功案例、边界案例、失败案例各 3-5 例
- **分析角度**：检索质量、提示词设计、模型能力
- **输出**：评估报告（含指标表格、案例记录、改进建议）

### 4.6 UI 层（Gradio）

- **布局**：顶部 Tab 切换（智能问答 / 文本摘要 / 知识问答）
- **输入区**：
  - 文本输入框（多行，自适应高度）
  - 参数调节滑块（Top-K 1-10、Temperature 0-1）
- **输出区**：
  - 主结果展示区域
  - 可折叠参考文档片段（含来源标注和相似度分数）
- **评估面板**（内置）：
  - 人工评分录入（正确性/完整性/流畅性下拉选择）
  - 评分结果实时显示

## 五、配置管理

系统采用 YAML 配置驱动，所有参数集中在 `config/config.yaml` 中管理：

```yaml
# LLM 配置
llm:
  provider: deepseek          # deepseek | mimo | qwen
  api_key: ${DEEPSEEK_API_KEY}
  base_url: https://api.deepseek.com/v1
  model: deepseek-chat
  temperature: 0.7
  max_tokens: 2048

# Embedding 配置
embedding:
  model_name: BAAI/bge-m3
  device: auto                # auto | cpu | cuda
  batch_size: 32

# 检索配置
retrieval:
  top_k: 5
  rerank_enabled: true
  reranker_model: BAAI/bge-reranker-v2-m3

# 分块配置
chunker:
  chunk_size: 512             # tokens
  overlap: 128                # tokens
  min_chunk_length: 20        # 字符

# 数据采集配置
data:
  arxiv_max_results: 20
  news_max_articles: 15
  qa_min_pairs: 200
```

## 六、技术选型总览

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| 语言 | Python ≥3.10 | 生态完善，AI/ML 工具链成熟 |
| Embedding | BGE-M3 | 多语言、多粒度，中英文检索SOTA |
| 向量库 | FAISS (CPU) | Meta开源，轻量高效，开发友好 |
| 重排序 | bge-reranker-v2-m3 | 与BGE-M3配套，效果最优 |
| LLM | DeepSeek API | 性价比高，中文能力强，OpenAI兼容 |
| LLM备选 | MiMo-v2.5-pro | OpenAI兼容接口，切换零成本 |
| UI | Gradio ≥4.0 | 快速搭建，支持多Tab，内置组件丰富 |
| 采集 | arxiv, requests | 成熟稳定，文档齐全 |
| 清洗 | BeautifulSoup4 | HTML解析标准库 |
| 去重 | datasketch (MinHash) | 高效近似去重，适合大规模文本 |
| 评估 | rouge-score, sacrebleu, bert-score | 学术标准三件套 |
| 配置 | PyYAML | 简洁可读，支持环境变量 |
| Token计数 | tiktoken | OpenAI开源，精确token计数 |
