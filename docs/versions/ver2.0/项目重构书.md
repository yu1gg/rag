## 1. 重构背景

当前项目已经完成了 RAG 问答、文本摘要、向量检索、评估等核心能力，但整体工程形态仍然是“单仓 Python + CLI + Gradio”的实验型结构：

- 根目录同时承载代码、配置、数据、文档与运行产物
- `main.py` 同时承担命令入口、流程编排、服务启动等多种职责
- 前端界面依赖 `Gradio`，适合演示，不适合后续产品化与页面扩展
- `src/*` 中已经形成较完整的算法模块，但缺少标准后端接口层
- 当前没有清晰的 `frontend / backend / docs` 分层，不利于协作、测试、部署和后续迭代

因此，Ver 2.0 的重构目标不是推倒重写算法能力，而是将现有成果迁移到更清晰、更标准、更接近真实项目交付的工程结构中。

## 2. 重构目标

本次重构以 `D:\8\2\tongshi_all_two-main` 为工程形态参考，但不照搬其业务内容，重点对标以下方面：

1. 根目录简化为 `frontend`、`backend`、`docs` 三大主体
2. 后端改为 `FastAPI` 服务化，提供标准 REST API
3. 前端改为 `Vue 3 + Vite`，替代 `Gradio`
4. 文档体系独立整理，架构文档、接口文档、部署文档分开维护
5. 现有 RAG 核心模块保留并迁移，不做无意义重写
6. 为后续测试、部署、扩展打好工程基础

本次重构完成后，项目应从“课程作业/实验项目形态”升级为“可继续开发的标准前后端分离项目形态”。

## 3. 重构原则

### 3.1 借鉴结构，不照搬业务

`tongshi_all_two-main` 的价值主要在于工程组织方式，而不是其课程平台业务本身。  
本项目应对标其：

- 目录分层方式
- FastAPI 后端组织方式
- Vue 前端组织方式
- 文档独立管理方式
- 配置、路由、服务、接口分层思路

不建议直接照搬其：

- 教师、课程、学生、管理后台等业务模块
- 过重的业务表结构
- 与本项目无关的权限体系

### 3.2 保留现有算法资产

以下内容已经具备复用价值，应迁移而不是重写：

- 数据采集、清洗、补充、分块
- 向量化与 FAISS 索引
- 检索与重排
- Prompt 模板与 LLM 调用
- 自动评估与人工评估

### 3.3 工程先清晰，再逐步增强

Ver 2.0 第一阶段要解决的是“结构不清晰”，不是一次性把所有能力堆满。  
因此应优先完成：

- 前后端拆分
- API 化
- 页面替换
- 文档归档
- 最基本的测试与运行链路

鉴权、用户系统、复杂后台、云存储、多租户等内容，不作为本轮必须项。

## 4. 当前结构存在的问题

结合当前仓库实际情况，主要问题如下：

### 4.1 根目录职责过多

当前根目录同时存在：

- `src/` 业务代码
- `config/` 配置
- `data/` 原始数据、索引和评估结果
- `docs/` 文档
- `main.py` 主入口

这种结构在项目刚开始时可行，但随着能力变多，会出现：

- 新成员难以快速理解项目边界
- 数据与代码混放，部署时容易混乱
- 前后端无法独立开发

### 4.2 `main.py` 过于集中

当前 `main.py` 同时承担：

- CLI 调度
- 数据流程编排
- 索引构建
- Gradio 启动
- 评估运行

这会导致入口文件越来越重，后续一旦增加 API 服务、后台任务、批处理脚本，维护成本会迅速上升。

### 4.3 UI 技术路线不利于继续扩展

当前 `src/ui/app.py` 基于 `Gradio`。  
它适合做 Demo，但不适合：

- 复杂页面布局
- 状态管理
- 多页面路由
- 前后端职责分离
- 与后端 API 长期演进

### 4.4 缺少标准服务边界

当前算法模块虽然拆得比较细，但调用关系主要还是“Python 内部直接互调”，缺少：

- 明确的 API 契约
- 标准请求/响应模型
- 面向前端的服务接口
- 可独立测试的后端边界

## 6. 技术栈对标方案

### 6.1 后端技术栈

建议采用：

- `FastAPI`
- `Uvicorn`
- `Pydantic`
- `python-dotenv`
- `SQLAlchemy`
- `Alembic`
- `openai`

- `FlagEmbedding`
- `faiss-cpu`
- `tiktoken`

### 6.1.1 为什么后端要改成 FastAPI

相比当前 `Gradio + Python 内部调用`，`FastAPI` 的优势在于：

- 天然支持前后端分离
- 自动生成接口文档
- 更适合封装问答、摘要、检索、评估等服务
- 更利于后续加鉴权、日志、测试、部署

### 6.1.2 数据库存储建议

本项目与 `tongshi_all_two-main` 不同，不需要一开始就做重数据库业务。  
因此建议：

- Ver 2.0 初期：保留文件存储为主，结构化元数据可使用 `SQLite + SQLAlchemy`
- 后续若有多用户、历史记录、任务管理需求，再切换到 `MySQL`

这样既对标了工程技术路线，又避免过度设计。

### 6.2 前端技术栈

建议采用：

- `Vue 3`
- `Vite`
- `TypeScript`
- `Vue Router`
- `Pinia`
- `Axios`
- `Element Plus`

### 6.2.1 为什么前端要替换 Gradio

重构后前端需要承担：

- 多页面导航
- 表单交互
- 对话结果展示
- 检索片段展示
- 参数控制
- 评估结果可视化

这些能力使用 `Vue 3 + Vite` 会明显优于 `Gradio`。

### 6.3 文档技术栈

文档继续使用 Markdown，但要按用途拆分：

- 架构文档
- API 文档
- 部署文档
- 版本方案文档

## 7. 后端重构设计

### 7.1 推荐分层

后端建议采用类似对标项目的分层方式：

```text
routes -> services -> rag modules -> storage
```

具体职责如下：

- `routes/`：接收 HTTP 请求，做参数校验，返回统一响应
- `schemas/`：定义请求体、响应体、DTO
- `services/`：编排业务流程
- `rag/`：承载原有算法核心模块
- `core/`：配置、异常、日志、响应格式
- `models/`：仅放确实需要入库的结构化实体

### 7.2 推荐后端目录细化

```text
backend/app/
├─ api/v1/routes/
│  ├─ health_routes.py
│  ├─ rag_routes.py
│  ├─ dataset_routes.py
│  ├─ index_routes.py
│  └─ evaluation_routes.py
├─ core/
│  ├─ config.py
│  ├─ response.py
│  ├─ exceptions.py
│  └─ logger.py
├─ schemas/
│  ├─ common.py
│  ├─ rag.py
│  ├─ dataset.py
│  ├─ index.py
│  └─ evaluation.py
├─ services/
│  ├─ rag_service.py
│  ├─ dataset_service.py
│  ├─ index_service.py
│  └─ evaluation_service.py
├─ rag/
│  ├─ data/
│  ├─ indexing/

│  ├─ generation/
│  └─ evaluation/
└─ models/
   ├─ chat_record.py
   ├─ dataset_record.py
   └─ task_record.py
```

### 7.3 推荐统一响应格式

为与对标项目风格保持一致，建议后端统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败时：

```json
{
  "code": 400,
  "message": "参数错误",
  "data": null
}
```

这样前端处理会更统一。

### 7.4 推荐第一批 API

建议优先落地以下接口：

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/v1/health` | 健康检查 |
| `POST` | `/api/v1/rag/qa` | 智能问答 |
| `POST` | `/api/v1/rag/summary` | 文本摘要 |
| `POST` | `/api/v1/rag/search` | 知识检索 |
| `GET` | `/api/v1/datasets/stats` | 数据集统计 |
| `POST` | `/api/v1/index/rebuild` | 重建索引 |
| `GET` | `/api/v1/index/status` | 查看索引状态 |
| `POST` | `/api/v1/evaluations/run` | 运行评估 |
| `GET` | `/api/v1/evaluations/latest` | 获取最新评估报告 |

### 7.5 离线脚本与在线 API 分离

当前 `collect / clean / chunk / index / eval` 都挂在一个 `main.py` 下。  
重构后建议拆成两部分：

- 在线服务：`backend/main.py`
- 离线脚本：`backend/scripts/*.py`

建议脚本如下：

- `collect_data.py`
- `clean_data.py`
- `chunk_data.py`
- `build_index.py`
- `run_evaluation.py`

这样接口服务不会和长耗时批处理混在一起。

## 10. 配置与存储重构

### 10.1 配置方案

建议改为“`.env` 为主，代码默认值兜底”的方式，接近对标项目：

示例：

```env
APP_ENV=dev
APP_HOST=127.0.0.1
APP_PORT=8050

OPENAI_API_KEY=your-api-key
OPENAI_BASE_URL=https://api.deepseek.com/v1
OPENAI_MODEL=deepseek-chat

EMBEDDING_MODEL=BAAI/bge-m3
RERANKER_MODEL=BAAI/bge-reranker-v2-m3

STORAGE_DIR=backend/storage
```

### 10.1.1 是否保留 YAML

建议策略如下：

- 环境相关配置：放 `.env`
- 少量固定参数：写在 `config.py` 默认值中
- 只有当 RAG 参数变得很多时，再保留一个 `rag.yaml`

这样更简洁。

### 10.2 存储方案

当前 `data/` 建议整体迁移至：

```text
backend/storage/
├─ raw/
├─ processed/
├─ index/
└─ eval/
```

其中：

- `raw/`：原始文档、QA 数据
- `processed/`：chunks 等处理中间产物
- `index/`：FAISS 索引与元数据
- `eval/`：评估结果

### 10.2.1 Git 管理建议

建议将以下内容加入忽略：

- 大体积模型缓存
- FAISS 索引文件
- 运行日志
- 临时评估结果
- 本地环境文件

## 13. 建议的实施顺序

为了避免“边迁边乱”，建议按下面顺序执行：

1. 先建新骨架，不急着删除旧代码
2. 先迁后端算法模块，再补 API
3. 前端先做最小可用页面，不追求一次性完整
4. 等 Vue 页面可用后，再退役 Gradio
5. 所有核心链路跑通后，再删除旧 `src/ui/app.py`
6. 最后再清理旧根目录中的 `config/`、`data/`、`src/`

这个顺序能最大程度降低重构中断风险。

## 14. 风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| 模型加载慢 | API 首次调用延迟高 | 使用懒加载和单例服务 |
| 索引重建耗时长 | 请求阻塞 | 先保留离线脚本方式，后续再考虑后台任务 |
| Gradio 到 Vue 切换期间功能不一致 | 重构体验变差 | 先保证问答、摘要、检索三条主链路可用 |
| 目录迁移导致导入路径混乱 | 开发效率降低 | 阶段 1 完成后统一命名与导入规范 |
| 过早引入复杂数据库 | 工期拉长 | 初期只保留必要元数据存储 |

## 15. 验收标准

Ver 2.0 重构完成后，至少应满足以下标准：

1. 根目录结构调整为 `frontend / backend / docs`
2. 后端能够通过 `FastAPI` 提供问答、摘要、检索接口
3. 前端能够通过 `Vue` 页面调用后端接口并展示结果
4. 现有 RAG 核心能力没有丢失
5. 数据、索引、评估结果已迁入 `backend/storage`
6. 文档中至少包含重构书、架构说明、接口说明、部署指南
7. 不再以 `Gradio` 作为项目主界面

## 16. 本次重构的结论

本项目非常适合做这次重构，因为它的算法层已经具备雏形，问题主要不在“能力缺失”，而在“工程组织方式还停留在单体实验阶段”。

因此，Ver 2.0 的最佳路线不是重写业务逻辑，而是：

- 以 `tongshi_all_two-main` 为工程结构参考
- 以 `FastAPI + Vue 3` 为技术路线
- 以“保留现有 RAG 核心，重构工程边界”为核心原则

最终目标是把当前项目升级为一个简洁、标准、可继续迭代的前后端分离项目。

## 17. 推荐下一步

第二轮完成后，下一步更合适的动作不再是“搭骨架”，而是继续完善新结构内部能力：
